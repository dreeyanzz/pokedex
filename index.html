  • Lets you browse through a collection with buttons
  • Shows additional information about each image
  • Has special effects (the silhouette mode)
  • Remembers what photo you're viewing
  • Can be turned on and off

Application Architecture (MVC Pattern)

This application follows the Model-View-Controller pattern, a professional architecture used in millions of applications:

Model: The Pokemon class - represents the data
View: JavaFX components - displays the data
Controller: Our methods in the App class - handles logic and user input

Why does this matter? This separation makes the code:

  • Easier to maintain: Change the UI without touching the data logic
  • Testable: You can test business logic separately from the UI
  • Reusable: The Pokemon class could be used in different applications
  • Scalable: Easy to add new features without breaking existing code
As you read through this code, try to identify which parts are Model (data), View (visual), and Controller (logic). This mental exercise will help you understand software architecture better!

2. Understanding Packages (In-Depth)

What is a Package?

Just like you organize files on your computer into folders (Documents ‚Üí School ‚Üí Java Projects), packages organize Java code into logical groups. This isn't just for neatness - it's essential for building real applications. 

package com.example;

This single line tells Java: "This file belongs to the com.example package."

Critical Rule: The package statement must be the VERY FIRST line in your Java file (comments don't count). Put ANYTHING else before it, and your code won't compile. No exceptions!

Why Packages Exist - The Real Reasons

Benefit Without Packages With Packages
Name Conflicts Only one "User" class allowed in entire project Can have com.facebook.User AND com.twitter.User
Organization All 1000+ classes in one giant folder Organized by feature/purpose
Access Control Everything is either totally public or private Package-private access for internal APIs
Distribution Conflicts with other libraries Your code safely coexists with others
Real Scenario: You're building a Pokemon game engine. Smart organization:
  • com.mycompany.pokemon.creatures - Bulbasaur, Charmander classes
  • com.mycompany.pokemon.battle - Battle system, attack calculations
  • com.mycompany.pokemon.ui - Pokedex UI, battle screen
  • com.mycompany.pokemon.data - JSON loading, save games
  • com.mycompany.pokemon.ai - Computer opponent logic
Now when you need to find the battle code, you know exactly where to look!

Package Naming Convention - Industry Standard

Java uses reverse domain notation. Here's the pattern:

You Own Package Name Example Class
example.com com.example com.example.App
myapp.org org.myapp org.myapp.Main
GitHub: username io.github.username io.github.username.Pokedex
School project edu.university.studentname edu.university.studentname.Project
NEVER start packages with capital letters (Com.Example). Packages must be all lowercase. This isn't optional - it's a universally followed Java convention. Breaking it marks you as a beginner.
Package names directly map to folder structures. If your package is com.example, your file MUST be in com/example/YourFile.java. Mismatch = compilation error!

Package Access Modifiers - A Hidden Feature

Besides public and private, Java has package-private (default) access: 

class HelperClass {  // No public keyword = package-private
    // Only classes in the SAME package can see this
}

This is incredibly useful for internal implementation details you don't want other packages accessing!

3. Understanding Imports (Complete Guide)

What Are Imports and Why Do We Need Them?

Java has thousands of pre-built classes (String, ArrayList, File, etc.). If Java loaded ALL of them every time, your programs would be incredibly slow. Instead, you explicitly tell Java which classes you need using import statements.

Think of imports like ingredients in a recipe. You don't bring your entire pantry to the kitchen - you only bring the flour, eggs, and sugar you actually need. Imports work the same way!

Import Syntax - The Two Forms

// Single class import (preferred)
import javafx.scene.control.Button;

// Wildcard import (use sparingly)
import javafx.scene.control.*;
Best Practice: Use specific imports (first form) rather than wildcards (*). Why? It makes your code clearer about what you're actually using, and prevents accidental name conflicts. IDEs like IntelliJ can auto-generate imports for you!

Our Imports - Detailed Breakdown

Jackson Library (JSON Processing)

import com.fasterxml.jackson.annotation.JsonProperty;
import com.fasterxml.jackson.core.type.TypeReference;
import com.fasterxml.jackson.databind.ObjectMapper;

What each does and why we need it:

  • ObjectMapper: The main JSON converter. Takes JSON text like {"name":"Pikachu"} and creates a Pokemon Java object. Without this, we'd have to manually parse JSON character by character (nightmare!)
  • JsonProperty: Handles naming differences. If JSON says "isLegendary" but your Java variable is "legendary", this annotation maps them correctly
  • TypeReference: Tells ObjectMapper we're reading a List<Pokemon>, not just one Pokemon. Java generics info is normally erased at runtime, but TypeReference preserves it
What Jackson Does For Us: 
// JSON file contains:
{"name":"Pikachu", "dexNumber":25, "types":["Electric"]}

// Jackson automatically converts it to:
Pokemon pikachu = new Pokemon();
pikachu.setName("Pikachu");
pikachu.setDexNumber(25);
pikachu.setTypes(Arrays.asList("Electric"));
Imagine doing this conversion manually for 151 Pokemon. Jackson saves us thousands of lines of code!

Forget an import? Java will show an error like "Cannot find symbol". This means Java doesn't know what class you're talking about. Solution: add the correct import statement!

Standard Java Imports

import java.io.IOException;
import java.io.InputStream;
import java.util.ArrayList;
import java.util.List;
  • IOException: Exception type for file operations. When file reading fails (file missing, permission denied), this error type is thrown
  • InputStream: Abstract stream for reading data. We use it to read Pokemon images and JSON data from files
  • ArrayList: Resizable array. Unlike regular arrays (fixed size), ArrayList grows/shrinks automatically as you add/remove items
  • List: Interface for list collections. We use List as the type, ArrayList as the implementation (flexibility!)
You don't need imports for classes in:
  • java.lang package (String, System, Math, etc.) - automatically imported
  • The same package as your class
  • Primitive types (int, double, boolean) - these aren't classes!

What Happens If You Forget Imports?

// Without imports:
Button btn = new Button();  // ERROR: Cannot find symbol 'Button'

// Manually using full name (verbose):
javafx.scene.control.Button btn = new javafx.scene.control.Button();

// With import (clean):
import javafx.scene.control.Button;
Button btn = new Button();  // ‚úì Works!

Imports are just shortcuts to avoid typing full package names everywhere!

4. Understanding Classes & Objects

From Primitives to Classes - The Big Leap

You know primitive types:

int age = 25;
double price = 19.99;
boolean isActive = true;

But what if you need to store complex information like a Pokemon? You'd need: 

String name = "Pikachu";
int dexNumber = 25;
String type1 = "Electric";
String type2 = null;  // No second type
int evolutionStage = 2;
// ... and more!
Managing 7+ separate variables for ONE Pokemon is messy and error-prone. What if you need 151 Pokemon? That's over 1000 variables! This is why we need classes.

Classes - Your Own Custom Data Types

A class is a blueprint for creating a new type of data: 

public class App extends Application {
    // All our code goes here
}
Class vs Object - The Cookie Cutter Analogy:
‚Ä¢ Class = Cookie cutter (the blueprint/template)
‚Ä¢ Object = Actual cookie (created from the template)

One cookie cutter can make hundreds of cookies. One class can create hundreds of objects!
Part Meaning Why It Matters
public Visible to everyone Other code can use this class
class This is a class definition Creates a new type
App Name of the class Must match filename (App.java)
extends Application Inherits from Application Gets JavaFX superpowers for free!
Pokemon Class Example: 
public class Pokemon {
    private String name;
    private int dexNumber;
    // ...
}
Now we can create Pokemon objects: 
Pokemon pikachu = new Pokemon();
pikachu.setName("Pikachu");
pikachu.setDexNumber(25);

Pokemon charizard = new Pokemon();
charizard.setName("Charizard");
charizard.setDexNumber(6);
Much cleaner than managing separate variables!

Inheritance - Building on Existing Code

extends Application means our App class inherits all capabilities of Application class.

Inheritance is like genetics: You inherit traits from your parents (eyes, height), but you're also your own person with unique traits. Similarly, App inherits Application's methods but adds its own unique features!

What we get from extending Application:

  • The start() method that JavaFX automatically calls
  • The launch() method that starts our app
  • All the infrastructure to create GUI applications
  • Event handling capabilities
Without extends Application, we'd have to build all this infrastructure ourselves - hundreds of lines of complex code. Inheritance lets us reuse existing, tested code!

Best Practices & Professional Tips

Code Organization

  • One public class per file: The public class name must match the filename
  • Helper classes as inner classes: Pokemon class is inside App because it's only used there
  • Group related methods: All setup methods together, all action methods together
  • Use meaningful names: updateUI() is better than doStuff()

Common Pitfalls to Avoid

Forgetting to initialize variables: Using List<Pokemon> list; without = new ArrayList<>() causes NullPointerException!
Ignoring exceptions: Empty catch blocks hide errors. Always at least print the error!
Not checking for null: Always check if files/resources loaded before using them!

Common Issues & How to Fix Them

Problem: "Cannot find symbol"

Cause: Missing import or typo in class name

Solution: Add the correct import statement or check spelling

Problem: "NullPointerException"

Cause: Trying to use an object that's null

Solution: Initialize the variable or check if (object != null) before using

Problem: Images/files not loading

Cause: Wrong file path or files not in resources folder

Solution: Ensure files are in src/main/resources/ and paths start with /

Problem: UI not updating

Cause: Forgot to call updateUI() after changing state

Solution: Always update the UI after changing data that should be displayed
