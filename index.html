Architecture

MVC Design Pattern

This application follows the Model-View-Controller architectural pattern, ensuring clean separation of concerns, maintainability, and testability. Each component has a distinct responsibility with well-defined interfaces between layers.

üì¶ Model Layer

File: PokedexModel.java

Responsibilities:

  • Manages application state and business logic
  • Loads and stores Pok√©mon data from JSON
  • Handles navigation (next/previous)
  • Manages reveal/hide state
  • Tracks power on/off state
  • Provides data access methods
  • Wraps around boundaries (first ‚Üî last)

Key Methods:

  • loadPokemonData() - JSON parsing
  • nextPokemon() - Navigate forward
  • previousPokemon() - Navigate backward
  • toggleRevealHide() - Toggle display mode
  • getCurrentPokemon() - Get active Pok√©mon
  • powerOn() / powerOff() - State control
üé® View Layer

File: PokedexView.java

Responsibilities:

  • Renders the user interface
  • Displays Pok√©mon information and sprites
  • Creates interactive button hitboxes
  • Applies visual effects (silhouettes)
  • Manages layout and positioning
  • Loads fonts and background images
  • Updates UI based on model state

UI Components:

  • Background image (pokedex.png)
  • Pok√©mon sprite display (ImageView)
  • Name, description, types labels
  • Dex number display
  • Interactive button rectangles
  • Custom pixel font (Press Start 2P)

Jackson Annotations: No special annotations are needed because field names match JSON keys exactly. Jackson uses reflection to map JSON properties to Java fields automatically.
PokedexApp.java - Application Entry Point

The main class that extends Application and initializes the MVC components. It sets up the Scene and Stage, then delegates control to the Controller. 

public class PokedexApp extends Application {
    @Override
    public void start(Stage primaryStage) {
        // Initialize MVC components
        PokedexModel model = new PokedexModel();
        PokedexView view = new PokedexView();
        PokedexController controller = 
            new PokedexController(model, view);
        
        // Create scene
        Scene scene = new Scene(
            view.getRoot(), 
            view.SCREEN_WIDTH, 
            view.SCREEN_HEIGHT
        );
        
        // Setup stage
        primaryStage.setTitle("Pok√©dex");
        primaryStage.setScene(scene);
        primaryStage.setResizable(false);
        primaryStage.show();
        
        // Initial display
        controller.updateView();
    }
}

JSON Data Loading

Jackson ObjectMapper handles the deserialization of JSON data into Java objects. The Model uses this to populate the Pok√©mon list at startup. 

public void loadPokemonData() throws IOException {
    ObjectMapper mapper = new ObjectMapper();
    InputStream jsonStream = getClass()
        .getResourceAsStream("/kanto_pokemon.json");
    
    if (jsonStream == null) {
        throw new IOException("Cannot find JSON file");
    }
    
    // Deserialize JSON array into List<Pokemon>
    pokemonList = mapper.readValue(
        jsonStream,
        new TypeReference<List<Pokemon>>() {}
    );
    
    currentIndex = 0; // Start at first Pok√©mon
}

Field Descriptions
Field Type Description
picture String Filename of the Pok√©mon sprite (must exist in resources/pokedex/)
name String Pok√©mon name (displayed prominently)
evolutionStage Integer Evolution stage: 1 (basic), 2 (stage 1), 3 (stage 2)
shortDescription String Brief description shown in the Pok√©dex screen
dexNumber Integer National Pok√©dex number (1-151 for Kanto)
isLegendary Boolean Whether the Pok√©mon is legendary
types Array[String] Pok√©mon types (e.g., ["Fire"], ["Water", "Flying"])
Data Validation Rules
  • Picture field: Must match an actual image file in resources/pokedex/
  • Evolution stage: Must be 1, 2, or 3 (no other values)
  • Dex number: Should be unique and sequential for Kanto (1-151)
  • Types: Can have 1 or 2 types (no Pok√©mon has zero types)
  • Short description: Should be brief enough to fit in the display area
Tip: Keep descriptions under 150 characters to ensure they fit properly in the UI. The font and text wrapping are configured for concise entries.

Responsive Behavior

Current Implementation: Fixed window size (548√ó400) for pixel-perfect display matching the background image.

Why Fixed Size? This ensures the UI remains aligned with the background graphics. Resizing would require dynamic scaling of all positioned elements, which would compromise the authentic retro aesthetic.

Extension Opportunity: For a responsive version, you could:

  • Scale all coordinates proportionally to window size
  • Use percentage-based positioning
  • Implement SVG graphics instead of PNG for scaling
  • Add multiple layout profiles for different screen sizes

Development Workflow

Adding New Pokemon
  1. Prepare the sprite image: Ensure it's a PNG file with transparent background, sized appropriately (square recommended)
  2. Add image to resources: Place the file in src/main/resources/pokedex/
  3. Update JSON data: Add a new entry to kanto_pokemon.json with all required fields
  4. Verify filename match: Ensure the picture field in JSON exactly matches the image filename
  5. Rebuild project: Run mvn clean install
  6. Test: Run the application and navigate to the new Pok√©mon to verify display
Modifying UI Layout
  1. Locate constants: Open PokedexView.java and find the positioning constants at the top
  2. Adjust coordinates: Modify SCREEN_X, SCREEN_Y, or other position/size constants as needed
  3. Recompile and test: Run the application to verify positioning looks correct
  4. Update button hitboxes: If buttons no longer align with graphics, adjust their coordinates
  5. Consider background changes: If modifying the background image, update all dependent coordinates accordingly
Pro Tip: Use graphic editing software to measure exact pixel coordinates on the background image before updating constants.
Adding New Features

Follow the MVC pattern:

  1. Model: Add state variables and business logic methods to PokedexModel.java
  2. View: Add new UI components and positioning in PokedexView.java
  3. Controller: Create event handlers and wire up interactions in PokedexController.java
  4. Test thoroughly: Verify all state transitions work correctly and UI updates properly

Example - Adding a "Favorite" feature:

  • Model: Add Set<Integer> favorites and toggleFavorite() method
  • View: Add a star icon/button and highlight for favorites
  • Controller: Add onFavoriteToggle() handler and update view to show favorite status

Java Module System Configuration

The project uses JPMS (Java Platform Module System). The module-info.java file declares module dependencies and accessibility: 

module com.pokedex {
    // Required modules
    requires javafx.controls;
    requires javafx.fxml;
    requires com.fasterxml.jackson.databind;
    
    // Export packages (make accessible to other modules)
    exports com.pokedex;
    
    // Open packages for reflection (needed for Jackson)
    opens com.pokedex.model to com.fasterxml.jackson.databind;
}
Important: The opens directive is critical for Jackson. Without it, Jackson cannot use reflection to access private fields in the Pokemon class, resulting in deserialization errors.

Best Practices

Code Organization

  • MVC Separation: Keep business logic in Model, UI in View, coordination in Controller. Never access View directly from Model or vice versa - always go through Controller.
  • Constants: UI coordinates and dimensions are defined as constants at the top of the View class for easy maintenance. Group related constants together.
  • Resource Management: Use ClassLoader for resource access to ensure portability between development and production environments. Always use getResourceAsStream() for files in the JAR.
  • Error Handling: Check for null resources and handle loading failures gracefully with try-catch blocks and user feedback.
  • State Validation: Always verify powered-on state before allowing interactions. Use guard clauses at the start of Controller methods.
MVC Best Practices
‚úÖ DO ‚ùå DON'T
Keep Model completely UI-independent - should work without JavaFX Import JavaFX classes in Model layer
Make View a passive component that only displays data Put business logic or state management in View
Have Controller mediate all communication between layers Let Model directly update View or View directly change Model
Return copies or immutable data from Model to prevent external modification Return direct references to internal collections in Model
Keep methods focused with single responsibility (one method does one thing) Create large multi-purpose methods that do many things
Security & Validation
  • JSON Validation: Validate JSON structure before deserialization to prevent parsing errors
  • Array Bounds: Check array bounds before accessing Pokemon list (especially with user-driven navigation)
  • Resource Existence: Handle missing image files gracefully with fallback images or error messages
  • Input Sanitization: If adding search functionality, sanitize user input to prevent injection issues
  • Data Validation: Validate evolution stage values (1-3), dex numbers (1-151), and ensure required fields exist
  • Type Safety: Use strong typing and avoid raw types or excessive casting
Performance Considerations

Current Implementation:

  • Lazy Image Loading: Images are loaded on-demand per Pokemon, not all at startup - saves memory
  • JSON Caching: JSON is loaded once at startup and cached in memory for the application lifetime
  • Effect Reuse: ColorAdjust effect is created once and reused for all silhouettes
  • Font Sharing: Custom font is loaded once and shared across all labels

Optimization Opportunities:

  • Image Caching: Cache loaded images to avoid reloading when navigating back to previously viewed Pokemon
  • Preloading: Preload adjacent Pokemon images in background for smoother navigation
  • Virtual Scrolling: If implementing a grid view, only render visible items
  • Thumbnail Generation: Create smaller thumbnails for list views to reduce memory usage
Code Quality Guidelines
  • Naming Conventions: Use descriptive names (onPowerOn not doThing), follow Java naming standards
  • Documentation: Add Javadoc comments to public methods explaining parameters and return values
  • Method Length: Keep methods short (ideally under 30 lines) - extract complex logic into helper methods
  • Magic Numbers: Avoid magic numbers in code - use named constants with descriptive names
  • DRY Principle: Don't Repeat Yourself - extract repeated code into reusable methods
  • Error Messages: Provide helpful error messages that guide users or developers to solutions
  • Logging: Use proper logging (java.util.logging) instead of System.out.println for production code
